# Daily Jarvis / Hermes Lab Note — 2026-07-05

## Today's focus

Sunday weekly review. Look back at the past week's daily notes, identify what was planned versus what was actually added, and set clear priorities for the coming week. The goal is not to add new features today but to make sure the existing direction is coherent and the next steps are concrete.

## Safe architecture note

A weekly review pass is itself an architectural practice. Before adding new modules or examples, it is worth checking whether the existing public contracts in `docs/daily/` and `examples/` still reflect the current direction. If a daily note described a future module idea that has since been partially built, the note should be treated as a reference point, not a specification. Public documentation should describe what exists, not what is hoped for.

The separation between public planning documents and private runtime configuration remains the core safety boundary. Weekly review is a good time to confirm that nothing from the private side has accidentally drifted into the public repo.

## Future module idea

A "module health summary" helper that reads the public module registry (once it exists as a JSON file) and flags entries that have not been reviewed in more than seven days. The helper would only inspect safe metadata fields such as `last_reviewed`, `status`, and `approval_level` — never private endpoints, tokens, or device addresses.

## Security / secret reminder

Weekly review is also a good time to run `public_safety_scan.py` over the repo to confirm that no high-signal risk patterns have been introduced. Do not commit API keys, tokens, OAuth files, webhook URLs, real IP addresses, device credentials, private chat logs, personal documents, payment data, or claims that an integration is production-ready before it has actually been built and verified.

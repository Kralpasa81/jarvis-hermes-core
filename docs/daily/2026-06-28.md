# Daily Jarvis / Hermes Lab Note — 2026-06-28

## Today's focus

Turn the recent "safe preview first" planning habit into a small reusable artifact. A module should be easy to describe publicly before it is connected to private adapters, real accounts, device state, documents, or delivery channels.

## Safe architecture note

A public module preview manifest is a lightweight boundary contract. It should name the module, explain its public purpose, show a mock command, list allowed synthetic inputs, list blocked private inputs, describe dry-run outputs, and state the approval level needed before any real action occurs. This keeps the repository useful for architecture review while keeping runtime credentials and personal context outside Git.

## Small actionable task

Add a tiny `examples/module_preview_manifest.py` helper that prints a synthetic manifest in JSON or Markdown. Use it when sketching future assistant features so each module starts with public-safe inputs, redaction expectations, and an explicit approval policy.

## Future module idea

Create a dashboard card that reads a collection of preview manifests and groups modules by status: documentation-only, mock-preview-ready, approval-required, private-runtime-only, and blocked. The card should use only synthetic sample data in this public repo.

## Security / secret reminder

Never commit API keys, tokens, OAuth files, webhook URLs, private chat logs, personal documents, payment details, real home-network addresses, device credentials, or unverified production claims. Public manifests should document safe contracts and blocked data types, not private runtime facts.

## Safe architecture note

A public module should be able to answer three boundary questions before implementation moves into a private runtime:

1. **What can be public?** Synthetic examples, schemas, dry-run previews, and generic documentation.
2. **What must stay private?** Credentials, OAuth files, chat IDs, real device identifiers, local file paths, private documents, and exact network details.
3. **What requires approval?** External messages, publishing, purchases, bookings, deletes, device control, or any workflow that changes real-world state.

Keeping these answers next to the module design prevents public docs from drifting into private operational details.

## Security / secret reminder

Do not commit API keys, tokens, OAuth exports, webhook URLs, chat IDs, home IP addresses, device credentials, private file paths, personal documents, payment data, or real assistant logs. Public examples should use synthetic data and must not imply that a real service, account, or device was contacted.

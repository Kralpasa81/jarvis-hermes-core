# Daily Jarvis / Hermes Lab Note — 2026-06-23

## Today's focus

Define a small safety-first planning checkpoint for Jarvis / Hermes workflows before they move from public examples into a private runtime.

The goal is to make every new module answer three simple questions: what it reads, what it can change, and when it must ask for approval.

## Safe architecture note

A useful module should keep its responsibilities narrow and visible:

1. **Input boundary** — public templates may describe generic data shapes, but private files, chats, device names, and account details stay outside the repo.
2. **Decision boundary** — the assistant can classify a request as read-only, preview-only, approval-required, or blocked before any action runs.
3. **Execution boundary** — private adapters should execute only after validation and, for risky tasks, explicit user confirmation.

This keeps public planning material helpful without turning it into a leak of real infrastructure or personal context.

## Security / secret reminder

Do not commit API keys, tokens, OAuth files, webhook URLs, private chat logs, device credentials, personal documents, private addresses, payment details, or real home-network information. Public examples should use synthetic placeholders only and must not claim production behavior that has not been built and tested.

## Today's focus

Move yesterday's module registry idea from a future note into a small public-safe prototype. The goal is to describe planned Jarvis / Hermes modules in a consistent registry without implying that private adapters, credentials, or real device integrations are already connected.

## Safe architecture note

A module registry should separate the public contract from the private runtime layer. Public entries can list synthetic inputs, preview outputs, approval level, status, and documentation gaps. Private adapters should remain outside the repository and hold any real endpoints, tokens, device names, account identifiers, personal paths, or automation credentials.

## Small actionable task

Add a dependency-free `module_registry_builder.py` example that prints a synthetic registry as JSON or Markdown. Keep the data generic, local-only, and suitable for dashboard or roadmap planning without reading environment variables or calling external services.

## Future module idea

Create a stale-module review helper that flags registry entries whose public contract, approval rule, or manual fallback has not been reviewed recently. The helper should only inspect safe metadata dates and never scan private runtime configuration.

## Security / secret reminder

Do not commit API keys, tokens, OAuth files, webhook URLs, real IP addresses, device credentials, private chat logs, personal documents, payment data, or claims that an integration is production-ready before it has actually been built and verified.

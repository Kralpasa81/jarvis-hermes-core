# Daily Jarvis / Hermes Lab Note — 2026-06-29

## Today's focus

Define a small "integration readiness" habit for future Jarvis / Hermes modules. Before a private adapter is connected, the public repo should be able to explain what the module will accept, what it will refuse, how it previews work, and when human approval is required.

## Safe architecture note

Treat each module as two layers: a public-safe planning layer and a private runtime adapter. The planning layer can include synthetic examples, schemas, checklists, and dry-run previews. The private adapter is the only layer that may know real account identifiers, device names, network details, document paths, or delivery channels, and it should stay outside this public repository.

## Small actionable task

Add a lightweight readiness review checklist to future module notes: purpose, allowed synthetic inputs, blocked private inputs, dry-run output, approval level, rollback/manual fallback, and a verification command. This keeps new modules reviewable without exposing secrets or implying production readiness.

## Future module idea

Create a dependency-free `integration_readiness_report.py` example that reads a small synthetic module description and prints a pass/warn/block summary. It should never call APIs or inspect local private configuration; it should only evaluate explicit public-safe fields.

## Security / secret reminder

Do not commit API keys, tokens, OAuth files, webhook URLs, private chat logs, personal documents, payment data, real home-network addresses, device credentials, or unverifiable production claims. If a detail is needed only at runtime, document its placeholder name and keep the real value in private configuration.

## Safe architecture note

A public assistant repository benefits from a two-step release habit:

1. **Write with synthetic data only** — examples should use placeholder module names, generic paths, and mock status payloads.
2. **Scan before publishing** — run a lightweight local check for token-like strings, credential assignments, private key blocks, sensitive filenames, and private-network URLs.
3. **Review claims manually** — do not describe private adapters, automations, metrics, or production integrations as complete unless they have actually been implemented and verified outside the public repo.

This keeps the public project useful while still respecting the boundary between public planning and private runtime configuration.

Do not commit API keys, tokens, OAuth exports, webhook URLs, chat IDs, home IP addresses, device credentials, private file paths, personal documents, payment data, or real assistant logs. A scanner can catch obvious patterns, but final responsibility still requires human review before publishing.

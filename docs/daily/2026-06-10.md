# Daily Jarvis / Hermes Lab Note — 2026-06-10

## Today's focus

Make the action-approval boundary easier to discuss before any private Telegram, smart-home, or document workflow is connected.

The practical goal is to keep public planning useful while making it obvious which assistant actions can run automatically and which ones must wait for explicit human confirmation.

## Safe architecture note

A Jarvis-style assistant should classify each action before execution:

- **read-only** actions can usually run without approval when they do not expose private data;
- **low-risk notifications** can run automatically with rate limits and non-sensitive wording;
- **external publishing, device control, deletion, booking, and payment-related actions** should require approval or stay blocked by default.

This boundary keeps the public repo focused on reusable safety patterns instead of private runtime wiring.

## Small task added

Added `examples/approval_matrix.py`, a dependency-free Python script that prints a public-safe approval matrix in Markdown or JSON.

It uses generic action categories only, does not call APIs, does not inspect local devices, and does not require tokens or configuration files.

Create a private approval gateway later that receives proposed actions from workflow modules, attaches a risk category, and asks the user for confirmation before any external post, device change, deletion, booking, or payment-adjacent step.

## Security / secret reminder

Never commit Telegram tokens, API keys, OAuth files, device credentials, payment details, private chat logs, personal documents, exact home-network addresses, or real approval transcripts.

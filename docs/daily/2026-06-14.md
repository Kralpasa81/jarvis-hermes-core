## Safe architecture note

Assistant actions should be reviewed as structured data before execution. A simple review object can keep the public planning layer separate from private runtime details:

1. **Action summary** — describe the user-visible intent without secrets or personal identifiers.
2. **State impact** — mark whether the action is read-only, notification-only, publishing, device control, or destructive/financial.
3. **Approval policy** — decide whether the action can run automatically, needs rate limiting, requires confirmation, or should be blocked.
4. **Safe preview** — show what would happen without including tokens, private paths, account IDs, or home-network details.

Keeping these fields explicit helps dashboard cards, Telegram flows, and future automation modules share one safety vocabulary.

## Safe architecture note

A command gateway should separate three steps:

1. **Intent preview** — classify a generic command into a module such as dashboard, documents, notifications, media, smart home, monitoring, or knowledge.
2. **Safety gate** — detect whether the request is read-only, preview-only, approval-required, strong-approval-required, blocked, or private-runtime-only.
3. **Execution adapter** — keep real files, accounts, device controls, chat delivery, and credentials outside the public repo until explicit approval rules are implemented and tested privately.

Public examples should show the route and approval expectation, not claim that real services were contacted.

## Small actionable task

Added `examples/safe_command_router.py`, a dependency-free script that accepts a generic command phrase and prints a route preview in Markdown or JSON. The output intentionally does not echo the full input command, which helps avoid copying accidental private details into public logs or screenshots.

Example safe commands:

```bash
python3 examples/safe_command_router.py --command "show dashboard status summary"
python3 examples/safe_command_router.py --command "draft a YouTube upload checklist" --format json
```

## Security / secret reminder

Do not commit API keys, tokens, OAuth files, webhook URLs, chat IDs, private IP addresses, device credentials, personal documents, payment data, real command logs, or screenshots that reveal private assistant context. Public routing examples should use synthetic phrases and must not execute real actions.

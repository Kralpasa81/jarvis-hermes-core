# Daily Jarvis / Hermes Lab Note — 2026-06-12

## Today's focus

Define a small reliability habit for Jarvis / Hermes modules: every public example and future private adapter should make its current state easy to inspect before it is allowed to trigger actions.

The practical goal is to keep the command-center concept observable, simple to debug, and safe to demo without exposing real accounts, device names, documents, or credentials.

## Safe architecture note

A useful assistant workflow should separate three kinds of information:

- **capability metadata** — what a module can do, documented with public-safe names and examples;
- **runtime status** — whether the module is available, degraded, or blocked, with private details redacted by the adapter;
- **action requests** — any state-changing operation that must pass through an approval rule before execution.

This makes dashboards and chat summaries helpful while avoiding the risky pattern of mixing secrets, live device state, and control commands in the same payload.

## Small actionable task

Add a lightweight readiness checklist for future modules before wiring them into the assistant gateway:

1. Confirm the module can run without hard-coded secrets.
2. Provide a dry-run or preview mode for public examples.
3. Document which outputs are safe to show in logs, screenshots, or chat.
4. Mark every action as read-only, approval-required, or blocked by default.
5. Keep real credentials and private endpoint details in ignored local configuration only.

## Future module idea

Create a `module_readiness` helper that accepts a public-safe module manifest and prints a short Markdown or JSON readiness report. It should be dependency-free, avoid network calls, and never require tokens or personal configuration.

## Security / secret reminder

Do not commit API keys, bot tokens, OAuth files, device credentials, private chat logs, personal documents, payment details, exact home-network addresses, or screenshots that reveal private runtime state.

## Safe architecture note

A Jarvis / Hermes module should be able to describe its boundary before it runs. A simple boundary card keeps the public planning layer separate from private runtime details:

1. **Module purpose** — explain the user-facing value in generic terms.
2. **Allowed public inputs** — accept only synthetic examples, labels, or non-sensitive summaries in public templates.
3. **Blocked data** — reject credentials, exact network addresses, personal documents, chat logs, and payment details.
4. **Approval rule** — document whether the module is read-only, notification-only, approval-required, or blocked.
5. **Preview behavior** — show what would happen without executing commands, calling APIs, or controlling devices.

This pattern helps future dashboard, OCR, media, and smart-home modules stay reviewable before they become operational.
